# Модуль Rendering

`Rendering/` отвечает за визуальное представление состояния симуляции. Здесь находятся рендерер, GPU-пайплайны, слои отрисовки, а также rendering benchmarks и тесты.

## Роль

- подготовка draw-данных
- работа с GPU-буферами
- настройка пайплайнов
- разбиение визуализации на слои
- отрисовка состояния мира на экран
- локальные benchmark-сценарии рендера

## Границы

- что здесь не должно жить:
- логика пользовательского интерфейса
- прикладное поведение приложения
- физические правила мира как часть render-layer логики

## Структура

Обычно этот модуль удобно воспринимать так:

- `Renderer` — общая координация рендера
- `Renderer2D` / `Renderer3D` — конкретные режимы камеры и матриц
- `layers/` — отдельные типы визуализации (атомы, бокс, визуализации)
- `pipelines/` — инициализация GPU-пайплайнов
- `benchmarks/` — benchmark-сценарии
- `tests/` — юнит тесты рендера

## Поток

Верхний уровень рендера сейчас построен вокруг `RendererWGPU`.

Упрощенно один кадр выглядит так:

```text
drawShot()
  -> выбор RenderData
  -> setupSceneUniforms()
  -> beginPass()
  -> drawWorldPass()
       -> atoms
       -> bonds
       -> grid
       -> field
       -> box
       -> memory order
  -> endFrame()
```

Практически это означает:

- `Renderer` собирает общий проход кадра
- общие uniform-данные сцены обновляются один раз на проход
- дальше вызываются специализированные части рендера для конкретных типов визуализации
- каждая часть сама готовит CPU-данные, обновляет нужные GPU-буферы и делает draw


## Причины

Такой подход выбран по нескольким причинам:

- общая логика прохода кадра лежит в одном месте и не размазывается по проекту
- все, что относится к общему состоянию кадра, проще держать внутри `Renderer`
- отдельные типы визуализации остаются локальными и не требуют переписывать весь проход кадра

## Паттерн

Обычно новый кусок визуализации проходит через три стадии:

```text
1. подготовить данные на CPU
2. загрузить или обновить буферы на GPU
3. выполнить draw
```

На уровне кода это обычно выглядит так:

- `prepare...CpuData(...)`
- `uploadPrepared...Gpu(...)`
- `drawPrepared...Gpu(...)`

Именно поэтому в `Renderer.h` сейчас есть методы вроде:

- `prepareAtomsCpuData(...)`
- `uploadPreparedAtomsGpu(...)`
- `drawPreparedAtomsGpu(...)`
- `prepareGridCpuData(...)`
- `uploadPreparedGridGpu(...)`
- `drawPreparedGridGpu(...)`

## Расширение

Если нужно добавить новый тип визуализации, обычно достаточно идти по такому пути:

1. Добавить данные состояния в `LayerState.h`, если нужны новые буферы, bind group или кэш.
2. Добавить pipeline в `PipelineState.h`, если нужен новый shader/pipeline.
3. Добавить инициализацию pipeline в `pipelines/`.
4. Добавить GPU-буфер или init-функцию в `Renderer`, если для этого типа нужен отдельный persistent buffer.
5. Добавить trio-методы по шаблону:
   - `prepare...CpuData(...)`
   - `uploadPrepared...Gpu(...)`
   - `drawPrepared...Gpu(...)`
6. Подключить вызов в `drawWorldPass()`.
7. Если нужно, добавить benchmark в `Rendering/benchmarks/`.

Минимальная практическая схема такая:

```text
данные мира
  -> prepareNewThingCpuData()
  -> uploadPreparedNewThingGpu()
  -> drawPreparedNewThingGpu()
  -> вызов из drawWorldPass()
```

## Модель

Этот слой отвечает не за то, что симулируется, а за то, как результат симуляции показать.

Даже если часть вычислений будет выполняться на GPU, это не значит, что рендер должен становиться местом, где живет модель мира. Вычислительный backend и логика визуализации — это разные оси архитектуры.

## Что Можно Улучшить

Сейчас рендер уже рабочий, но в нем остаются направления для дальнейшего упрощения и выравнивания архитектуры.

Наиболее естественные кандидаты:

- сделать пайплайны более универсальными, чтобы уменьшить количество почти одинаковой инициализации под близкие типы геометрии
- выделить более явный абстрактный слой `Layer`, чтобы типы визуализации были описаны единообразно
- сократить дублирование в prepare/upload/draw цепочках там, где шаги структурно одинаковы
- еще жестче отделить общую координацию кадра от логики конкретных визуальных слоев

При этом важный критерий остается тем же: улучшение абстракции не должно превращать рендер в систему со скрытыми регистрациями, тяжелыми фабриками или неочевидным жизненным циклом объектов.
