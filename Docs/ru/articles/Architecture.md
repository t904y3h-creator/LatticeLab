# Архитектура

Этот документ описывает проект в виде крупных блоков: где находится движок `LATTICE`, как разделены ответственность и как идут данные от симуляции до экрана.

## Схема

```text
LatticeLab:

			+------------------+
			|       App        |
			| UI, окна, ввод   |
			+------------------+
		       /            \
		      /              \
			 V                V
+-------------------+   +------------------+
|  Lattice Engine   |   |    Rendering     |
| симуляция, физика |   | GPU, слои, draw  |
+-------------------+   +------------------+
```

На практике `App` управляет сценой и настройками, `Lattice` обновляет физическое состояние, а `Rendering` получает подготовленные данные для отображения.

## Принципы

- простые и явные решения предпочтительнее скрытых регистраций, сложных фабрик и неочевидной инициализации
- `App`, `Lattice` и `Rendering` должны иметь четкие границы ответственности
- физическая модель не должна смешиваться с логикой отрисовки
- рендер и benchmark-код должны быть удобны для изолированного запуска и измерений
- архитектура должна позволять развивать проект по частям, а не требовать глобального рефакторинга на каждое изменение

## Модули

### `App/`

Статья по модулю: [Приложение](./App.md)

Здесь лежит прикладной слой:

- окна
- UI
- viewport
- обработка пользовательского ввода
- glue-код между симуляцией и рендером

Это точка входа для пользователя и место, где собирается поведение приложения как desktop-программы.

### `Lattice/`

Статья по модулю: [Движок Lattice](./Lattice.md)

Это ядро проекта.

Обычно сюда стоит идти, если нужно понять:

- как устроена физическая модель
- как хранятся данные симуляции
- как меняется состояние мира между шагами
- какие данные отдаются наружу для рендера или benchmark-сцен

### `Rendering/`

Статья по модулю: [Рендер](./Rendering.md)

Этот модуль отвечает за визуальную часть:

- общий `Renderer`
- 2D и 3D варианты камеры и матриц
- слои отрисовки
- GPU-пайплайны
- rendering benchmarks

Рендер получает уже подготовленные представления данных и преобразует их в набор GPU-команд.

## Поток

```text
1. App создает и настраивает сцену
2. Lattice обновляет состояние симуляции
3. Формируются данные для отображения
4. Rendering подготавливает CPU/GPU данные
5. Renderer вызывает draw
6. Картинка попадает на экран
```

Для benchmark-сцен схема почти та же, только без полного UI-слоя.

## Границы

```text
App:
- управляет приложением
- знает про UI и viewport

Lattice:
- знает про физику и состояние мира

Rendering:
- знает про буферы, пайплайны и draw
```

Что важно держать в голове:

- `Lattice` не должен превращаться в UI-слой
- `Rendering` не должен хранить физику
- `App` не должен тащить в себя детали GPU
