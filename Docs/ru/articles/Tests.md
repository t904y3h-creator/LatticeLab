# Юнит Тесты

Эта статья описывает тестовую инфраструктуру проекта: где лежат тесты, как они собираются и как добавлять новые проверки.

## Роль

Тесты нужны для изолированной проверки модулей без полного пользовательского сценария приложения.

В текущей структуре они решают две основные задачи:

- проверка вычислительного ядра `Lattice`
- проверка локальной математики и базового поведения `Rendering`

## Структура

Сейчас в проекте есть два тестовых слоя:

- `Lattice/tests/` — тесты движка
- `Rendering/tests/` — тесты рендера

Оба слоя подключаются через `CTest`.

На верхнем уровне это выглядит так:

```text
include(CTest)
  -> Lattice/tests/
  -> Rendering/tests/
```

Для `Rendering/tests/` есть дополнительный флаг:

- `BUILD_RENDERING_TESTS`

По умолчанию он включен.

## Текущие Тесты

### `LatticeTests`

Собирается из:

- `Lattice/tests/test.cpp`

Сейчас там есть проверки, связанные с:

- построением `Octree`
- распределением заряда по узлам
- приближением дальнодействующего кулоновского взаимодействия

Тестовый стиль здесь простой: без внешнего test framework, с локальными `expect(...)` и завершением процесса при ошибке.

### `RenderTests`

Собирается из:

- `Rendering/tests/CameraTests.cpp`

Сейчас там есть проверки для:

- центрирования камеры относительно bounds сцены
- базовой геометрии `RenderRay`
- пересечения луча со сферой

Это тоже легковесный executable без тяжелого тестового фреймворка.

## Сборка

Тесты подключаются через обычную конфигурацию проекта.

Обычно достаточно:

```bash
cmake --preset debug
cmake --build --preset debug
```

После этого можно запускать `CTest` в каталоге конкретного preset-а.

Пример для debug:

```bash
ctest --test-dir build/debug --output-on-failure
```

Если нужен только один тест:

```bash
ctest --test-dir build/debug -R LatticeTests --output-on-failure
ctest --test-dir build/debug -R rendering.module --output-on-failure
```

## Как Добавить Тест

Текущий стиль проекта предполагает очень прямой путь.

### Для `Lattice`

1. Добавить проверку в `Lattice/tests/test.cpp`, если это близкий по теме тест.
2. Если файл становится слишком большим, вынести новый `*.cpp` и добавить его в `Lattice/tests/CMakeLists.txt`.
3. Подключить нужные заголовки ядра.
4. Сделать тест воспроизводимым и не зависящим от `App` или `Rendering`.

### Для `Rendering`

1. Добавить новый `*.cpp` в `Rendering/tests/`.
2. Подключить его в `Rendering/tests/CMakeLists.txt`.
3. Проверять локальную математику, камеру, подготовку данных или другие части, не требующие полного UI-цикла.

## Практические Правила

- тест должен проверять одну понятную вещь
- если проверка требует полного приложения, это уже плохой кандидат для модульного теста

## Что Можно Улучшить

Текущая система уже рабочая, но ее можно развивать дальше:

- разбить крупные тестовые файлы на более мелкие тематические
- добавить больше проверок для `World`, `NeighborList`, `ForceField`, `RenderData`
- унифицировать локальные helper-функции и стиль сообщений об ошибках
- при необходимости позже перейти к более формальному test framework, если объем тестов заметно вырастет

Сейчас же плюс текущего подхода в том, что он очень простой, прозрачно собирается и не добавляет лишней инфраструктурной сложности.
