# Модуль Lattice

`Lattice/` — это ядро симуляции. Здесь живут данные мира, физическая модель, шаги обновления состояния и связанные с этим представления.

## Роль

- хранение состояния симуляции
- частицы, связи и структуры данных мира
- физические расчеты
- обновление состояния между шагами
- подготовка данных, которые затем могут использовать рендер или benchmark-сценарии

## Границы

- что здесь не должно жить:
- UI и логика окон
- логика draw и GPU-пайплайнов
- детали прикладного слоя desktop-приложения

## Независимость

`Lattice/` нужно воспринимать как самостоятельную библиотеку, а не как внутренний модуль приложения.

Это означает:

- движок должен собираться и работать без `App/`
- движок должен собираться и работать без `Rendering/`
- любая зависимость ядра от UI, viewport, `ImGui`, `WGPU` или других прикладных подсистем считается нарушением архитектурной границы

Практический критерий здесь простой: если убрать весь desktop-слой и весь рендер, `Lattice` все равно должен оставаться рабочим вычислительным модулем со своими `CLI`, benchmark- и test-сценариями.

Если вычисления со временем будут перенесены на GPU, это не меняет роли модуля: он все равно остается уровнем модели и симуляции.

## Структура

На верхнем уровне модуль разбит на несколько частей:

- `Engine/` — основное ядро симуляции
- `Generators/` — генерация стартовых сцен и конфигураций
- `CLI/` — консольный интерфейс к движку
- `benchmarks/` — производительные сценарии и измерения
- `tests/` — тесты модуля

Практически это значит:

- если нужно понять модель мира, идти в `Engine/`
- если нужно быстро создать сцену, смотреть `Generators/`
- если нужен вход в движок без GUI, смотреть `CLI/`
- если нужна производительность, смотреть `benchmarks/`

## Engine

`Engine/` — главный технический центр модуля.

Ключевые сущности здесь такие:

- `Simulation` — верхний уровень управления симуляцией
- `World` — отдельный мир со своим состоянием
- `physics/` — силы, интегратор, связи, vector field
- `NeighborSearch/` — spatial grid и neighbor list
- `metrics/` — профилирование, энергия, статистика
- `io/` — состояние симуляции и экспорт в `xyz`

### `Simulation`

`Simulation` — это управляющий уровень движка.

Он отвечает за:

- управление несколькими мирами
- выбор активного мира
- запуск шага симуляции
- удобный API для внешнего кода
- операции уровня приложения: запись `xyz`, смена параметров, batch-операции

Именно `Simulation` чаще всего видят `App`, `CLI` и benchmark-код.

### `World`

`World` — это конкретное состояние одного мира.

Внутри него находятся данные мира и текущая конфигурация расчета:

- атомы
- связи
- spatial grid  (пространственная сетка)
- neighbor list  (список соседей)
- выбранный интегратор и связанные с ним параметры
- выбранная конфигурация сил и связанных параметров
- vector field
- метрики и runtime state

То есть `World` лучше воспринимать не как набор разрозненных подсистем, а как объект состояния мира с данными симуляции и выбранной конфигурацией вычисления.

Если `Simulation` — это оболочка уровня управления, то `World` — это уже конкретный объект модели, который обновляется на каждом шаге.

## Physics

Подпапка `Engine/physics/` содержит физическую часть движка.

Ключевые области:

- `Integrator` — схема интегрирования и связанные параметры
- `ForceField` — вычисление взаимодействий
- `Bond` / `BondTable` — работа со связями
- `VectorField` — поле, связанное с анализом или визуализацией состояния

Это место, где определяются правила изменения системы.

## Neighbor Search

Подпапка `Engine/NeighborSearch/` отвечает за поиск соседей и пространственные структуры.

Здесь лежат:

- `SpatialGrid`
- `NeighborList`

Это критическая часть производительности движка, потому что без нее вычисление взаимодействий быстро становится слишком дорогим.

## Generators

`Generators/` содержит заготовки сцен и процедурную генерацию стартовых конфигураций.

Типичные задачи:

- заполнить мир частицами
- создать кристаллическую структуру
- создать газовую сцену
- быстро подготовить benchmark- или demo-конфигурацию

Этот слой полезен тем, что отделяет "создание стартового мира" от логики самого движка.

## CLI

`CLI/` — это отдельный консольный вход в движок.

Подробная статья: [CLI.md](./CLI.md)

Он нужен для случаев, когда не нужен UI и хочется:

- вручную гонять симуляцию
- запускать шаги из терминала
- смотреть базовые метрики
- быстро проверить поведение движка вне desktop-приложения

По текущему коду CLI умеет, например:

- `step`
- `run`
- `positions`
- `metrics`
- `reset`

То есть это небольшой интерфейс без GUI поверх `Simulation`.

## Benchmarks

`Lattice/benchmarks/` отвечает за измерение производительности движка.

Там есть две основные части:

- инфраструктура запуска и оформления результатов
- сами benchmark-сценарии в `physics/`

Типичные benchmark'и измеряют:

- шаг симуляции
- rebuild neighbor list
- rebuild spatial grid
- compute forces
- сортировку атомов
- post-processing стадий интеграции

Это важно архитектурно, потому что модуль движка проектируется не только как "правильный", но и как измеряемый.

## Tests

`Lattice/tests/` содержит тесты модуля.

Этот слой нужен для изолированной проверки ядра:

- корректности логики симуляции
- поведения ключевых структур и алгоритмов
- стабильности изменений без запуска UI и полного рендера

Идея простая: движок должен проверяться как самостоятельный модуль, а не только через приложение целиком.

## Поток

Типичный поток работы через движок выглядит так:

```text
create Simulation
  -> create World
  -> fill world with atoms / bonds / generators
  -> configure integrator and forces
  -> update()
  -> read state / metrics / derived data
```

На уровне приложения это обычно означает:

- `App` создает `Simulation`
- сцена или генератор наполняет мир
- `Application` вызывает обновление
- `Rendering` получает представление состояния мира

## Вход

Если нужно быстро войти в код движка, обычно полезно идти так:

1. `Engine/Simulation.h`
2. `Engine/World.h`
3. `Engine/physics/`
4. `Engine/NeighborSearch/`
5. `Generators/`
6. `CLI/main.cpp`
7. `benchmarks/physics/`

## Что Можно Улучшить

Сейчас ядро уже рабочее, но в нем тоже есть естественные направления развития архитектуры.

Наиболее интересные:

- более гибкая система вычислительного пайплайна, чтобы шаг симуляции можно было собирать из более явных стадий
- более явная конфигурация вычислительного пайплайна: какие силы, какие стадии и в каком порядке участвуют в шаге
- более модульная система физических и вычислительных подсистем, чтобы новые блоки подключались без разрастания центральной логики

Главный критерий здесь тот же, что и у всего `Lattice`: развитие абстракций не должно размывать независимость ядра и не должно тащить его в сторону UI- или render-зависимого дизайна.
