# Расширяемость системы

в этой статье описаны основные принципы и соглашения написания гибкого расшияемого приложения

Новая функциональность должна добавляться через новые модули, а не через изменение существующего кода.

```
Создать модуль
↓
Зарегистрировать
↓
Система обнаруживает его автоматически
↓
GUI и Lua получают доступ автоматически
```

Без правок ядра.

## Что считается модулем

Модуль это отдельная единица функциональности, которая:

- реализует известный контракт ядра
- регистрируется в общей системе регистрации
- может быть обнаружена автоматически
- предоставляет метаданные для GUI, сцены и скриптового слоя

Примеры:

```
Integrator
Thermostat
Force Field
Render Layer
Generator Primitive
Scene Loader
```

Ядро не должно знать конкретные реализации модулей.
Ядро должно знать только их интерфейсы, точки регистрации и правила жизненного цикла.

## Регистрация модулей

Регистрация должна происходить не через ручные if/switch, а через передачу в регистратор структуры описания модуля.

Базовая идея:

```
Создается структура параметров модуля
↓
Структура передается в регистратор
↓
Регистратор сохраняет фабрику и метаданные
↓
GUI, сцены и scripting layer используют уже зарегистрированное описание
```

Это позволяет:

- отделить реализацию модуля от механизма его обнаружения
- передавать метаданные вместе с фабрикой
- использовать один и тот же контракт для GUI, scene layer и runtime

### Пример структуры регистрации

```cpp
struct GeneratorRegistration {
    GeneratorFactory factory;
    GeneratorMetadata metadata;
};
```

```cpp
struct GeneratorMetadata {
    std::string id; // имя ищется по id в словаре переводов
    std::string category;
    std::vector<ParameterMetadata> parameters;
};
```

Где:

- `factory` - функция создания реализации
- `metadata` - описание параметров, доступных GUI и scene layer

### Пример регистрации

```cpp
registry.registerGenerator({
    .factory = []() { return std::make_unique<RandomFillGenerator>(); },
    .metadata = RandomFillGenerator::describe(),
});
```

### Пример использования со стороны системы

После регистрации система должна уметь:

- показать модуль в GUI
- создать форму параметров по `metadata`
- разрешить использование в scene description
- создать экземпляр через `factory`

То есть добавление нового модуля должно выглядеть так:

```cpp
registry.registerGenerator(makeRandomFillRegistration());
registry.registerGenerator(makeLatticeFillRegistration());
registry.registerGenerator(makeLoadPdbRegistration());
```

Без правок:

```cpp
Simulation
World
IOPanel
Lua API
```

Если для нового модуля нужно добавлять ветку в существующий switch, регистрация сделана неправильно.
## Производительность

Абстракции допускаются на границах систем.

Примеры:

```
Simulation Backend
Integrator
Thermostat
Renderer
Plugin API
```

Горячие вычислительные участки должны оставаться конкретными и максимально простыми.

Примеры:

```
Particle Storage
Neighbor Lists
SIMD Kernels
Force Calculation
Spatial Structures
```

Расширяемость не должна ухудшать производительность вычислительного ядра.

## Главный принцип

Ядро должно задавать правила взаимодействия модулей, но не ограничивать то, какие системы могут быть построены поверх него.

## Границы ядра

Ядро должно содержать:

```
Simulation
World
Particle Storage
Neighbor Search
Force Evaluation
Batch Operations
Module Contracts
Module Registry
```

Ядро не должно содержать:

```
Жестко зашитые генераторы сцен
Жестко зашитые сценарии модов
GUI-логику конкретных модулей
Скриптовую предметную логику
```

Если новая возможность добавляется через if/switch в Simulation, World, GUI или scripting layer, это признак слабой расширяемости.

## Слой описания сцен

Описание сцены должно быть отделено от вычислительного ядра.

Сцена должна представляться как данные:

```
World Definition
Generator Invocation
Region Definition
Composition Definition
Simulation Settings
```

Язык описания сцены может быть любым:

```
JSON
Lua
Binary Format
Custom DSL
```

Но итогом работы этого слоя всегда должна быть одна и та же внутренняя структура данных сцены.

Это позволяет:

- менять формат описания сцены без изменения ядра
- поддерживать несколько способов authoring
- автоматически строить GUI поверх метаданных модулей
- сериализовать и загружать сцены независимо от скриптового слоя

## GUI и скриптовый слой

GUI и scripting layer не должны содержать вычислительную логику ядра.

Они должны:

- выбирать модуль
- задавать параметры
- запускать модуль
- показывать доступные зарегистрированные возможности

Если для добавления нового primitive generator нужно вручную переписывать GUI или скриптовый API, значит система все еще недостаточно модульна.

## Масштаб расширения

Система должна поддерживать несколько уровней расширения.

### Уровень 1

Новая реализация существующего интерфейса.

Примеры:

```
Новый интегратор
Новый термостат
Новый потенциал
Новый render layer
```

### Уровень 2

Новая подсистема.

Примеры:

```
Новый solver
Новый storage backend
Новый render backend
```

### Уровень 3

Новая модель симуляции.

Примеры:

```
Classical Molecular Dynamics
Nuclear Physics
Quantum Computing
Cellular Automata
Continuum Simulation
```

Ядро не должно ограничивать появление новых моделей.

## Практическое правило

Перед добавлением новой функции нужно ответить на три вопроса:

1. Это новая реализация существующего контракта?
2. Это новая подсистема с отдельным контрактом?
3. Это новый слой поверх уже существующих primitive modules?

Если ответ "нет" и требуется менять существующее ядро без расширения через контракт, архитектура выбрана неправильно.
